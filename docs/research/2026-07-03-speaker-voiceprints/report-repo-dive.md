# Voiceprints Repo Dive

FluidAudio is pinned at 0.15.4 (`Package.swift:11-12`, `Package.resolved:13-18`). Current shipped diarization uses only FluidAudio offline APIs: `OfflineDiarizerManager(config:)`, `OfflineDiarizerModels.defaultModelsDirectory()`, `prepareModels(directory:)`, `process(audioURL:) -> DiarizationResult`, and optional `OfflineDiarizerConfig.withSpeakers(...)` constraints (`Sources/MacParakeetCore/Services/Diarization/DiarizationService.swift:50-61`, `73-90`, `202-213`). There are no source/test calls to `SpeakerManager` or `extractSpeakerEmbedding`; `SpeakerManager` appears only in ADR future-work text (`spec/adr/010-speaker-diarization.md:27`, `170`, `229`).

## Execution and Modes

Exact diarization return point: `DiarizationService.diarize(audioURL:)` (`Sources/MacParakeetCore/Services/Diarization/DiarizationService.swift:101-157`). It lazy-prepares models, wraps `manager.process(audioURL:)` in `ANEInferenceGate.shared.withExclusiveAccess`, catches `OfflineDiarizationError.noSpeechDetected`, sorts `DiarizationResult.segments`, normalizes FluidAudio IDs (`speaker_0`) to stable `S1`/`S2`, builds `SpeakerInfo(id:"S1", label:"Speaker 1")`, and returns `MacParakeetDiarizationResult` (`104-116`, `124-157`).

File/URL/podcast: `TranscriptionService.transcribeAudio` converts input to a temp WAV, runs selected-engine STT, then if `shouldDiarize()` and words exist calls `diarizationService.diarize(audioURL: wavURL)` (`Sources/MacParakeetCore/Services/TranscriptionService.swift:1397-1447`). It merges by overlap via `SpeakerMerger.mergeWordTimestampsWithSpeakers` and stores speaker fields (`1448-1458`; merger at `Sources/MacParakeetCore/Services/Diarization/SpeakerMerger.swift:7-12`, `55-57`). Dictation is not diarized.

Meetings: finalization transcribes mic/system sources separately (`TranscriptionService.swift:1221-1264`, job `.meetingFinalize`). If enabled and a system track exists, only the isolated system WAV is diarized in `diarizeMeetingSystemIfNeeded` (`1266-1324`), then speaker IDs are namespaced to `system:S1` and labels to `Others 1` (`1292-1313`). `MeetingTranscriptFinalizer.finalize` keeps mic words as `speakerId = "microphone"`, optionally refines system words with diarization, then builds `speakers` and `diarizationSegments` (`Sources/MacParakeetCore/Services/MeetingRecording/MeetingTranscriptFinalizer.swift:23-85`, `87-99`, `103-159`). Insertion point for voiceprint matching: file path immediately after `diarResult` returns and before `SpeakerMerger.merge...` (`TranscriptionService.swift:1446-1458`); meeting path after `diarizeMeetingSystemIfNeeded` has mapped IDs and before `MeetingTranscriptFinalizer.finalize` (`1099-1109`). If matching changes labels only, update `speakers`; if it changes attribution, do it before merge/finalizer output is persisted (`1141-1146`, `1690-1694`).

## Data Model and Rename

Speaker labels live on one table: `transcriptions`. Columns: `wordTimestamps TEXT`, `speakerCount INTEGER`, `speakers TEXT`, and `diarizationSegments TEXT` (`Sources/MacParakeetCore/Database/DatabaseManager.swift:112-127`, `180-184`; model fields at `Sources/MacParakeetCore/Models/Transcription.swift:18-31`, `164-199`). `wordTimestamps` stores per-word stable `speakerId`; `speakers` is JSON `[{"id":"S1","label":"Speaker 1"}]`; `diarizationSegments` stores segment `speakerId/startMs/endMs` (`spec/01-data-model.md:119-123`, `170-175`). `rawTranscript`/`cleanTranscript` are plain text, not the speaker-label source of truth.

Rename today is structured-only. In the transcript detail speaker overview, tapping a label swaps in a `TextField`; submit/blur calls `commitSpeakerRename()` (`Sources/MacParakeet/Views/Transcription/TranscriptResultView.swift:2872-2920`, `2950-2990`). `TranscriptionViewModel.renameSpeaker` edits the matching `SpeakerInfo.label` and calls `TranscriptionRepository.updateSpeakers` (`Sources/MacParakeetViewModels/TranscriptionViewModel.swift:1241-1252`; repo write at `Sources/MacParakeetCore/Database/TranscriptionRepository.swift:433-439`). It does not rewrite `WordTimestamp.speakerId`, `diarizationSegments`, `rawTranscript`, or `cleanTranscript`.

## Audio Availability

Meetings create an app-managed session folder under `AppPaths.meetingRecordingsDir/<sessionID>/` (`Sources/MacParakeetCore/Services/MeetingRecording/MeetingRecordingService.swift:386-409`) with `microphone.m4a`, `system.m4a`, and `meeting.m4a` (`Sources/MacParakeetCore/Audio/MeetingAudioStorageWriter.swift:44-67`), plus optional cleaned mic (`MeetingRecordingOutput.swift:7-16`, `55-77`). Successful stop returns these paths and does not delete them; completion only deletes the recovery lock (`MeetingRecordingService.swift:535-688`, `691-697`). DB `filePath` points to mixed `meeting.m4a`; `meetingArtifactFolderPath` preserves the folder (`TranscriptionService.swift:1055-1067`, `416-420`; migration `DatabaseManager.swift:998-1008`). Archived retranscribe requires the source files (`MeetingRecordingOutput.swift:90-133`).

Audio can disappear by policy or UI. Meeting retention modes are `keepForever`, `deleteAfterDays`, `deleteImmediately`; fresh audio is saved unless `deleteImmediately` (`AppRuntimePreferences.swift:39-66`, `88-93`, `693-733`). Settings exposes “Meeting audio retention” and warns that removing audio disables playback/retranscription (`Sources/MacParakeet/Views/Settings/SettingsView.swift:1929-2023`). “Remove Audio Only” removes standard meeting audio files including cleaned mic and clears `filePath` while preserving `meetingArtifactFolderPath` (`Sources/MacParakeetCore/Utilities/TranscriptionAssetCleanup.swift:32-40`, `88-131`; UI/view-model entry `TranscriptionViewModel.swift:772-795`, bulk `TranscriptionLibraryViewModel.swift:283-299`, `362-424`).

Local file transcription stores the original path (`TranscriptionService.swift:341-343`, `579-582`) and the app does not own/delete `.file` audio (`TranscriptionAssetCleanup.swift:58-72`); re-embedding depends on the original file still existing. URL/podcast downloads are kept only when `shouldKeepDownloadedAudio() && persistResult`, otherwise temp media is cleaned after transcription (`TranscriptionService.swift:908-929`, `971-984`, `1493-1497`); `saveTranscriptionAudio` defaults true and is wired from settings (`AppEnvironment.swift:338-339`, `AppRuntimePreferences.swift:592-613`).

## Intent and Designer Constraints

Specs say current identity is anonymous and per-transcription: no cross-file speaker identity (`spec/02-features.md:1371-1380`, `spec/06-stt-engine.md:531-604`, `spec/adr/010-speaker-diarization.md:220-229`). Newer planning names the right product gap as speaker memory, with opt-in local profiles, suggestions requiring confirmation, and deletable sensitive voice data (`docs/plans/2026-06-14-002-speaker-diarization-world-class-architecture.md:21-45`, `197-225`, `422-431`, `506-529`). Current meeting docs stress source-first attribution: mic=`microphone`, system=`system`, optional diarization only refines system side (`docs/research/meeting-dual-stream-transcription-pipeline.md:303-324`, `396-399`).

Concurrency/model conventions: `TranscriptionService`, `DiarizationService`, and `STTScheduler` are actors; STT has interactive/background slots, with meeting finalize outranking file transcription in the background slot (`Sources/MacParakeetCore/STT/STTScheduler.swift:20-27`, `835-867`). Diarization is outside the STT scheduler and must use `ANEInferenceGate` for CoreML/ANE work on macOS 14 (`Sources/MacParakeetCore/Services/ANEInferenceGate.swift:3-31`, `54-71`; `DiarizationService.swift:107-114`). Model prep is lazy but onboarding/preflight accounts for diarization assets when speaker detection is enabled (`DiarizationService.swift:160-178`; `Sources/MacParakeetViewModels/OnboardingViewModel.swift:618-629`, `714-788`). Speaker detection is opt-in: `speakerDiarizationKey`, default `false`, Settings row “Speaker detection” (`AppRuntimePreferences.swift:501`, `612-614`; `SettingsViewModel.swift:422-425`, `716-724`; `SettingsView.swift:1357-1361`).
