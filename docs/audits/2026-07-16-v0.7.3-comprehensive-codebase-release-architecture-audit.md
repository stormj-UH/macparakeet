## 1. Executive verdict

### Code verdict

The audited main commit is a credible **0.7.3 code candidate** after the
dependency, build-tooling, and canonical-documentation repairs on this audit
branch. No reviewed defect in persistence, privacy, capture ownership, meeting
settlement, engine routing, or the public CLI contract requires a 0.8 release
or a broad architectural rewrite.

### Shipping verdict

**Superseded by the post-release publication record above.** The audit branch
itself did not publish; the final release workflow subsequently rebuilt from
the merged CLI-installer revert, signed and notarized the app and DMG, verified
the installed candidate and bundled CLI, and published one immutable artifact
through R2, Sparkle, and GitHub.

### Architecture verdict

The architecture is healthy and has several legitimately deep modules. The
best next improvements are concentrated, not structural churn:

1. Separate speech-model lifecycle policy from STTRuntime.
2. Make explicit microphone selection authoritative instead of silently
   falling through to another input.
3. Create one read-only Speech Memory query and citation meaning for Library,
   CLI, and future agent adapters.
4. Add a current-toolchain Swift concurrency gate and fix the newly exposed
   Sendable seam.
5. Deepen transcript-detail and Settings presentation by behavior domain,
   without mechanically splitting large files.

### Release-candidate CLI installer finding and disposition

MacParakeet has bundled `macparakeet-cli` inside the signed app at
`MacParakeet.app/Contents/MacOS/macparakeet-cli` since before 0.7.3. Issue #595
did not report a missing binary: its author explicitly showed the full bundled
path and requested a convenience alias or symlink so the shorter command would
resolve in Terminal.

PR #775 added an app menu item, Settings state, service, mocks, and tests to
create `/usr/local/bin/macparakeet-cli`. Its tests used writable temporary
directories. The PR correctly left signed-app `/usr/local/bin` permission QA
as a manual release gate, but that gate was not exercised before merge.

The first signed and notarized 0.7.3 candidate exposed the defect during
installed-app QA. On a normal macOS installation, `/usr/local/bin` was owned by
`root:wheel` and not writable by the app process. Clicking Install failed with
the user-visible error that the app did not have permission to save
`macparakeet-cli` in `bin`. The rejected candidate was build
`20260716214441`; its DMG was 148,933,259 bytes with SHA-256
`bda4d3e5087d7bc156cd4f5c9fd1fde9b332a9ac838305c09a334cd115d7ee26`.
It was never uploaded, added to the appcast, or published on GitHub.

The QA machine also had a separately installed Homebrew CLI at
`/opt/homebrew/bin/macparakeet-cli`, pointing into Homebrew's Cellar. That
installation is valid and independently versioned. On the machine's PATH it
would take precedence over an app-created `/usr/local/bin` link, so merely
adding administrator authorization would not make the feature unambiguous.

An initial repair proved that the app could detect the Homebrew location,
protect Homebrew-owned links, and request narrowly scoped administrator
authorization. Focused tests passed, but the repair was deliberately not
retained. It added privileged process execution and package-manager coexistence
policy for a convenience requested by one advanced user who already knew the
bundled path.

This was primarily an issue-triage and product-boundary oversight. Issue #595
was a reasonable convenience request, but it was treated as an app defect and
closed by implementation before deciding whether a GUI app should own shell
PATH configuration. The implementation then validated its writable-directory
contract in unit tests without validating the real privileged destination
before merge. The lesson is not to add broader PATH discovery or more mocks:
future shell-integration requests should first distinguish a missing bundled
capability from optional user-environment configuration, decide ownership, and
exercise any privileged filesystem assumption in installed signed-app QA.

The final product boundary is simpler:

- MacParakeet owns and verifies the CLI bundled inside its signed app.
- Homebrew owns its standalone CLI and `/opt/homebrew` links.
- The shell user owns PATH, aliases, and manually created links.
- The app does not modify shell profiles, write package-manager paths, or offer
  a privileged PATH-link installer.

PR #827 merged the removal before release. Issue #595 was closed as not planned
with the bundled-path and ownership explanation; release-readiness issue #824
was closed with the correction that its earlier installer-ready claim had been
superseded. This preserves the useful request history without representing the
removed convenience surface as a shipped fix.

The release fix therefore reverts PR #775 in full, including the menu item,
Settings row, installer service, and their dedicated tests. It does not remove
or change the bundled CLI, the public CLI contract, or the standalone Homebrew
distribution. The rejected candidate must be rebuilt, signed, and notarized
from the merged revert before publication. Release notes must not claim that
the CLI can be installed from Settings.

## 5. Subsystem review

### Capture and microphone routing

The shared microphone and meeting code now has substantially better ownership,
route-attempt diagnostics, prewarming, and teardown tests. PR #801 correctly
restores System Default as an implicit macOS route rather than coercing it to
the built-in microphone.

One policy remains intentionally called out for later work:
meetingInputDeviceAttempts tries a selected input, then System Default, then the
built-in microphone. That makes capture more likely to start, but a named input
can silently become a different input. This is a user-intent and privacy
contract, not merely an implementation detail.

### Meeting lifecycle, recovery, and AEC

The reviewed stop/start/finalize paths have bounded state transitions, recovery
artifacts, and explicit settlement. PRs #810 and #814 address credible failure
modes in the old stop path. AEC and cleaned-microphone artifacts are included
in strict packaging and checksum gates.

No code-level blocker was found. Real-call validation remains mandatory because
echo cancellation quality and hardware routing cannot be proven by unit tests.

### STT runtime and engine scheduling

STTRuntime and STTScheduler are the correct central control plane for engine
ownership, warm-up, leases, and inference ordering. PRs #783 and #813 improve
role clarity rather than duplicating runtimes.

The main architectural pressure is model lifecycle: download, availability,
cache layout, deletion, and telemetry are spread across static runtime/engine
methods and several GUI/CLI callers. FluidAudio 0.15.5 removing DownloadUtils
made that wide coupling visible.

### Persistence and user data

GRDB repositories remain useful seams, roughly one per table. DatabaseManager
is large because it is the chronological migration ledger; splitting it would
make data evolution harder to audit. No unsafe destructive recovery behavior,
unparameterized filename update, or migration-order defect was found.

Meeting artifacts and the database continue to be treated as user data.

### Text processing

The shared pipeline correctly centralizes ordered dictation/transcription work.
There is no evidence for creating another pipeline or splitting it by caller.
The recent multilingual and spacing changes are bounded by focused behavior
tests.

### CLI and agent-facing contract

The bundled CLI reports contract version 3.0.0 and its help/version paths work.
The public CLI has an independent semantic contract from the GUI app version,
so MacParakeet 0.7.3 plus CLI 3.0.0 is coherent.

The unsigned standalone CLI emits a nonfatal NSUserDefaults suite warning. The
installed, signed application path remains the authoritative release smoke.

### Update, distribution, and trust

The appcast feed, embedded public key, minimum OS, bundled helpers, legal
notices, model/runtime checksum, and matching dSYM were checked. The build
script now diagnoses Xcode's missing git-submodule helper and borrows the shell
Git helper directory when safe.

### Privacy, telemetry, and diagnostics

The local-first boundaries remain intact. Cloud transforms, downloads, update
traffic, and telemetry are explicit surfaces. A targeted scan of the nine
tracked diagnostic logs found no credential, token, prompt, transcript, email,
user-path, or URL disclosure matching the audit patterns.

Raw logs still should not be committed as routine debugging practice. Prefer
private attachments or explicitly redacted/synthetic fixtures.

### Documentation and plans

The canonical docs had lagged the product: they still called 0.6.24 stable and
described several shipped v0.7 capabilities as pending. The audit branch
corrects stable/candidate framing, shipped auto-stop/VAD/live-preview behavior,
and recent plan status. The capability-registry plan itself is complete for
Phase A; later phases remain separate gated work.

## 8. Release findings fixed on the audit branch

### 8.1 FluidAudio dependency range

Package.swift allowed FluidAudio 0.15.4 up to, but excluding, 0.16.0.
Package.resolved happened to contain 0.15.4 at revision
b9d43724cbdb5a980e441fd54180964e94d470f7. Version 0.15.5 replaces DownloadUtils
with ModelHub, while MacParakeet still invokes DownloadUtils in Nemotron and
other lifecycle paths.

The range could therefore resolve to a source-incompatible patch on a fresh
release build. The audit branch exact-pins 0.15.4. This is a release
stabilization measure, not a permanent refusal to upgrade. ModelHub migration
belongs in the speech-model lifecycle work.

### 8.2 Xcode missing git-submodule

The release machine reports Xcode 26.4.1 and Swift 6.3.1. With GIT_EXEC_PATH
unset, xcrun Git cannot run git submodule because Xcode's Apple Git installation
does not contain the git-submodule helper. Homebrew Git does contain it at:

**/opt/homebrew/opt/git/libexec/git-core/git-submodule**

This previously surfaced as an opaque xcodebuild package-resolution failure:
“Couldn't update repository submodules.”

The bundle script now:

1. Tests Xcode Git's submodule support.
2. Discovers the active shell Git's exec path.
3. Verifies that directory actually contains git-submodule.
4. Exports the verified helper path for xcodebuild.
5. Re-tests support and fails with an actionable error if repair is impossible.

The distribution preflight and troubleshooting docs now record this behavior.
A replay with GIT_EXEC_PATH explicitly removed passed the strict bundle.

### 8.3 Canonical release and feature truth

At audit time, README/spec/ADR surfaces were corrected to say:

- v0.7.2 is the current public stable release.
- The post-v0.7.2 reliability train is a 0.7.3 candidate.
- Meeting auto-stop, VAD chunking, cleaned-mic finalization, Cohere support, and
  live dictation preview shipped in the v0.7 line with their actual flags and
  defaults.
- The recent merged implementation plans are no longer described as open.

After publication, the canonical README/spec release blocks were advanced to
stable 0.7.3. The statements above remain the dated pre-publication checkpoint.

## 12. Architecture improvement portfolio

### A. Speech-model lifecycle module — strong, highest priority

**Evidence**

- STTRuntime is about 2,661 lines.
- It mixes inference ownership with static download, availability, deletion,
  cache layout, and telemetry behavior.
- EngineSettingsViewModel, SettingsViewModel, OnboardingViewModel,
  ModelsCommand, STTClient, and TranscriptionViewModel reach through different
  lifecycle implementations.
- FluidAudio 0.15.5's ModelHub change has a wide caller radius.

**Direction**

Create one lifecycle module that owns the model catalog, availability,
download, deletion, disk layout, operation telemetry, and quiescence needed for
disk mutation. Keep inference, leases, scheduling, warm-up, and active sessions
in STTRuntime/STTScheduler.

**Payoff**

- One policy for GUI and CLI.
- Smaller upstream-upgrade radius.
- Executable cache-mutation concurrency.
- Focused lifecycle tests.
- Less responsibility in STTRuntime without undoing ADR-016.

### B. Authoritative microphone intent — strong, hardware gated

System Default and a named microphone are different intents. Preserve System
Default as macOS's implicit route. When a user names an input, either open that
input or return a typed visible failure; do not silently record from a room mic.
Expose the effective route during capture and make preparation, Input Test,
start, recovery, and diagnostics share the same policy.

### C. Read-only Speech Memory query module — strong, strategic

Current search meanings differ:

- Library uses TranscriptionRepository.fetchLibraryPage.
- Dictation history uses DictationRepository.search.
- CLI search uses SegmentRepository.search and cited FTS segment hits.
- CLI history exposes additional transcription/dictation keyword paths.

Build one read-only module over the existing repositories and derived segment
index. It should own query interpretation, local filtering, ranking, context
budgeting, and citation provenance. GUI, CLI, and future agent integrations
should adapt to that shared meaning. It should not own LLM providers, chat, or
mutation.

Reconcile the tentative Context Engine and Meetings Workspace plans before
implementation.

### D. Current-toolchain concurrency boundary — strong engineering hardening

Make service protocols that cross actors explicit about Sendable and isolation.
Start with CommandLineToolInstalling and the Settings caller exposed by Swift
6.3.1. Audit ScreenCaptureKit completion closure annotations next. Add a
release-machine/current-Xcode CI lane or scheduled gate while keeping the
pinned reproducible release lane.

### E. Transcript-detail composed panes — worth exploring

TranscriptResultView is about 3,879 lines with more than forty local state
values spanning media, transcript editing/search, speakers, summaries, prompt
results, chat, retranscription, export, copy, and deletion.

Keep one composition screen but deepen coherent reader, timeline/speaker,
generated-results, chat, and media/export panes. Extend existing view-models
where they already own behavior. A file split that merely moves the same shared
state is not an architecture improvement.

### F. Settings domain composition — worth exploring

- SettingsView: about 3,921 lines
- SettingsViewModel: about 1,855 lines
- SettingsViewModelTests: about 3,124 lines
- SettingsRootViewModel: 97 lines

Make the root compose capture, engines/models, system/permissions,
storage/privacy, and integrations. Move persistence and behavior together one
domain at a time with compatibility forwarding. Do not create one view-model
per settings row.

## 13. Deliberate non-candidates

The following modules pass the deletion test and should not be reorganized just
because they are large:

- **DatabaseManager:** the chronological migration ledger is easier to audit in
  one place.
- **STTRuntime/STTScheduler control plane:** central inference ownership is the
  right depth. Extract lifecycle policy, not per-engine runtimes.
- **Meeting settlement/finalization:** another service would duplicate the
  current recovery and ownership seam.
- **Text-processing pipeline:** it already centralizes ordered shared behavior.
- **AppEnvironment:** it is a composition root; blindly splitting it would
  distribute construction side effects.
- **Repository interfaces:** they are useful user-data/test seams. Add Speech
  Memory query semantics above them.
- **Capability registry:** Phase A already exists and is complete. Do not
  rediscover it as missing architecture.

## 16. Release publication sequence

The final release followed this order after the maintainer approved
publication:

1. Select the exact approved Git SHA and record it.
2. Confirm origin/main still contains reviewed PR #822 at or after 6599603c.
3. Build once from the approved SHA with strict asset requirements.
4. Developer-ID sign all nested code and the application.
5. Create the DMG, notarize it, and staple it.
6. Copy/install into /Applications and run the complete installed-app smoke.
7. Hash the final immutable DMG and record its byte length.
8. Generate the Sparkle signature from that exact file.
9. Upload that one DMG.
10. Update and deploy the appcast with the matching version, build, URL, length,
    and signature.
11. Attach the same DMG to GitHub release v0.7.3 for Homebrew consumers.
12. Verify the remote DMG length/hash, appcast signature/version, GitHub asset,
    and update discovery from a prior stable install.

Do not rebuild between signing, Sparkle signing, GitHub upload, and appcast
publication. Those surfaces must all describe the same bytes.

Completed result: all three public surfaces describe build `20260717011712`
and the same 148,896,323-byte DMG. Sparkle verification succeeded against full
downloads from both R2 and GitHub. The broader physical scenarios in section
15 remain explicitly unclaimed, as noted in the post-release record.

## 19. Final handoff

The correct near-term posture is:

- Keep the release number at **0.7.3**.
- Preserve the narrow dependency/build-script/docs fixes.
- Do not pull architecture refactors into the release.
- Treat the Swift 6.3 Sendable finding as the first post-release engineering
  hardening item unless it can be fixed and independently verified without
  destabilizing the candidate.
- Treat merged PR #822 as part of the candidate and repeat its Settings visual
  checks in the signed app.
- Complete the signed physical matrix.
- Publish one immutable artifact through every distribution surface.

The strongest first post-release architecture slice is a speech-model lifecycle
module small enough to move FluidAudio from DownloadUtils to ModelHub without
touching every caller. The strongest correctness slice is authoritative
microphone intent with physical route verification. The strongest strategic
slice is a shared local, read-only Speech Memory query and citation contract.

Broad rewrites, a 0.8 version jump, or splitting large files for appearance
would add risk without solving the verified problems.
